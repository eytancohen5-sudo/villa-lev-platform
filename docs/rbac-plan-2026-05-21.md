# RBAC plan — villa-lev-platform

## 1. Context

Eytan needs to share write access with a small, growing set of named humans (co-founder, accountant, bankers, partners) without making the model world-writable and without breaking the unauthenticated investor / pitch / share-link read path. Today auth is a single hardcoded email in `src/lib/firebase.ts:29` (`ADMIN_EMAILS = ['eytancohen5@gmail.com']`) and a sibling `isAdmin()` rule in `~/Desktop/Villa Lev Claude/villa-lev-admin/firestore.rules:10-14` keyed off `request.auth.token.email`. The DB audit (`docs/db-and-export-audit-2026-05-21.md` §A.2 #1, #3) already flagged the email-hardcoded gate and the two-app rules race as the top open issues — this RBAC effort is the long-term resolution. Desired outcome: 3 roles in a `users/{uid}` collection, role-aware Firestore rules, a `useAuth()` that returns `role`, an `admin/team` UI to invite and manage humans, and a migration order that never logs Eytan out.

Hard constraints: no server actions, no route handlers, no middleware, no Cloud Functions if avoidable, static export must keep building, banker share-link reads of `/investor` and `/pitch` must remain unauthenticated.

## 5. Step-by-step plan

### Step 1 — ADR + roles decision (S)

`adr-writer` produces `docs/adr/0002-rbac-roles-and-users-collection.md`. Verify: ADR committed, plan-challenger sign-off.

### Step 2 — `users/` and `invites/` Firestore schema (S)

`users/{uid}`:
```
{
  uid: string (== doc id),
  email: string (lower-cased),
  displayName: string | null,
  role: 'admin' | 'editor' | 'viewer',
  createdAt: number (ms),
  invitedBy: string (uid) | null,
  lastSignInAt: number (ms) | null
}
```

`invites/{inviteId}` — `inviteId = lowercased(email)`:
```
{
  email: string (lower-cased, == doc id),
  role: 'admin' | 'editor' | 'viewer',
  invitedBy: string (uid),
  invitedAt: number (ms),
  note: string | null
}
```

**Decision: client-side role read from `users/{uid}`, NOT custom claims.** Custom claims require a Cloud Function (admin SDK only sets them), which violates the "no Cloud Functions if avoidable" constraint. The read-on-auth-change cost is one extra `onSnapshot` per session; Firestore charges one document read per session start and zero on cached re-renders. Acceptable.

### Step 3 — Firestore rules update in admin repo (M, critical path)

In `villa-lev-admin/firestore.rules` add helpers above the existing block:

```
function isLegacyAdmin() {
  return request.auth != null
    && request.auth.token.email != null
    && request.auth.token.email.lower() == 'eytancohen5@gmail.com';
}

function userDoc() {
  return get(/databases/$(database)/documents/users/$(request.auth.uid)).data;
}
function userExists() {
  return request.auth != null
    && exists(/databases/$(database)/documents/users/$(request.auth.uid));
}
function hasRole(role) {
  return userExists() && userDoc().role == role;
}
function isAdmin() {  // new behaviour: doc-based OR legacy bootstrap
  return hasRole('admin') || isLegacyAdmin();
}
function canEdit() {
  return isAdmin() || hasRole('editor');
}
function canView() {
  return canEdit() || hasRole('viewer');
}
```

Add new collection blocks:
- `users/{uid}`: read if `isAdmin() || request.auth.uid == uid`; create/update/delete if `isAdmin()`. Plus self-update carve-out limited to `lastSignInAt`.
- `invites/{inviteId}`: read if `isAdmin() || (request.auth != null && request.auth.token.email.lower() == inviteId)`; create/delete if `isAdmin()` OR same-email self-claim (`request.auth.token.email.lower() == inviteId`); update: forbidden.
- Also allow `create` on `users/{uid}` if there is a matching `invites/{userEmail}` doc whose role matches the new doc's role AND `request.auth.uid == uid` AND `request.auth.token.email.lower() == request.resource.data.email`.

Rewrite `scenarios/{scenarioId}` create/update predicate to `canEdit()` plus the existing shape validation. Read stays `if true` (public share-link).

**Read-latency / cost.** Every scenarios-write rule eval now does one `get(users/{uid})`. Firestore bills it as one extra doc read per write. At low single-digit saves/hour, negligible. No rate-limit risk at current write volume.

### Step 4 — Bootstrap Eytan's user doc (S, critical path)

**Chosen approach: transitional rule fallback over manual seed.** The `isLegacyAdmin()` predicate stays alive through the migration. Eytan can sign in and write the very first `users/{eytanUid}` doc himself from the new `admin/team` UI — no Firebase Console hand-creation. Once that doc exists with `role: 'admin'`, the legacy predicate is redundant; Step 8 removes it.

Implementer must NOT delete `ADMIN_EMAILS` from `src/lib/firebase.ts` in this step. Remains the safety net during migration.

### Step 5 — `useAuth()` extension + `userProfile.ts` helper (M)

Extend `AuthResult`:
```
{ user, role, isAdmin, canEdit, canView, loading, signIn, signOut, profile }
```

Where:
- `role` is `'admin' | 'editor' | 'viewer' | null` from `users/{uid}`, OR `'admin'` synthesised when `user` is set and `isAdminEmail(user.email)` is true (legacy bridge — drop at Step 8).
- `isAdmin = role === 'admin'`. Backwards compat.
- Module-level singleton gains a second `Unsubscribe` for the user-doc snapshot.

`doSignIn` no longer rejects non-allow-list emails. After `signInWithPopup`, call `claimInvite(user, db)`. If the user has no `users/{uid}` doc AND no matching invite (and is not legacy admin), sign them back out with "not authorised — ask Eytan for an invite."

### Step 6 — Onboarding flow / `claimInvite()` (S)

Pure function in `userProfile.ts`. Sequence inside one Firestore batched write:
1. `getDoc(users/{uid})` — if exists, return existing profile (idempotent re-sign-in).
2. `getDoc(invites/{email.lower()})` — if missing, return null (caller signs out).
3. Batched: `set(users/{uid})` from invite fields + `delete(invites/{email.lower()})`.

### Step 7 — UI gating + `admin/team` page (M)

- `src/app/admin/assumptions/page.tsx` line 1592 — `canWrite = canEdit`.
- All other `src/app/admin/*` pages — wrap in `if (loading) return spinner; if (!canView) return SignInCTA;`.
- `src/app/admin/team/page.tsx` (new) — gated to `isAdmin`. Lists `users/*`, invite form, per-row role-edit, per-row remove.
- `src/app/investor/page.tsx`, `src/app/pitch/page.tsx` — untouched. Still anonymous.

### Step 8 — Tighten rules / remove legacy fallback (S)

After Eytan's `users/{uid}` doc is verified live and one editor has signed in successfully:
1. Remove `isLegacyAdmin()` from rules — `isAdmin()` becomes `hasRole('admin')` only.
2. Remove `ADMIN_EMAILS` and `isAdminEmail()` from `src/lib/firebase.ts`. Legacy-synthesis branch in `useAuth.ts` deleted.

## 6. Test plan

- **Firestore rules unit tests** (`@firebase/rules-unit-testing`): scenarios.create as viewer denied; scenarios.create as editor allowed with valid shape; scenarios.create as editor denied with `size() > 50`; users.write as editor denied; users self-stamp `lastSignInAt` allowed; invites.create as editor denied; invites.delete by matching-email session allowed; `claimInvite` batched create+delete allowed only when invite exists and email matches.
- **Vitest unit tests** for `roleAtLeast`, `claimInvite` (mock Firestore), and the `useAuth` reducer.
- **Manual smoke**: anonymous loads `/investor` (200); anonymous loads `/admin/dashboard` (sign-in CTA); editor signs in and clicks save on assumptions (succeeds); viewer signs in and confirms save button is disabled.

## 7. Risks & open questions

**Biggest risk: a half-deployed rules update locks Eytan out.** The admin repo and platform repo each deploy to the same project; whichever ran `firebase deploy --only firestore:rules` last wins. Mitigation: this plan only modifies the admin-repo rules file (platform-side `firestore.rules` was already removed). Deploy must be coordinated from the admin repo.

**Step most likely to bite: Step 5.** Extending `useAuth()` while preserving the legacy synthesis branch and not breaking SSR. The module-level singleton in `useAuth.ts:42-99` requires reference-stable snapshots; adding a second async data source (the user-doc snapshot) means `current` state mutates twice on every sign-in. State shape is `{user, profile, loading}` with profile=`undefined` while pending vs `null` if known-missing. The `loading` flag must remain `true` until BOTH the auth state AND the profile snapshot have fired at least once for the current user.

**Open questions for Eytan:**
- Does the co-founder need `admin` or just `editor`? Defaulting to `editor` until specified.
- Should `users/{uid}` carry a `notify` boolean for future email notifications? Defaulting to no — cheap to add later.
- Invite emails are sent manually (Eytan messages the invitee) — no automated email since that would force a Cloud Function.
