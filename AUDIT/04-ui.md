# Villa Lev Platform — UI & Design System Audit

Generated: 2026-06-03
Auditor: UI / Design Systems specialist (read-only)
Scope: Admin view (`/admin/*`), Bank view (`/bank`, `/bank/optima`), Pitch deck (`/pitch`), shared component atoms.

---

## 0. Executive Overview

The platform has a coherent design language — a warm Mediterranean palette, a clear semantic token system in `globals.css`, and consistent use of `font-mono` for numeric data. That foundation is solid. The primary risks for the bank-facing view are:

1. **Chart hex values are not tokenised** — all Recharts stroke/fill colors are hardcoded hex strings, bypassing the design system. A palette change would require touching every chart individually.
2. **The DSCR chart's data-key labels are semantically reversed** — "Conservative" maps to the realistic scenario and "Realistic" maps to upside. A banker reading the legend will be misled.
3. **The tab strip in the bank view uses Tailwind's generic `slate` palette** rather than the brand token system — the only place in the entire bank view that leaks system palette.
4. **Recharts figures have no ARIA labels or `role="img"` wrappers** — they are invisible to screen readers, which matters for accessibility-conscious bank recipients.
5. **`KPICard` hardcodes two hex values** for positive/warning tones instead of the semantic `positive`/`warning` tokens — a minor but real divergence in the one shared atom.
6. **The admin-only "Connected viewers" section in `/bank`** contains three bare English strings not run through `t()`, making them untranslatable and amateurish if a Greek banker happens to see them.

The bank view overall reads as institutional: the term-sheet grid, coverage-ratio cards, and P&L table are structured and data-dense in the right way. The issues above are fixable without structural redesign.

## 3. Color System Assessment

### What works
The `globals.css` `@theme` block is a clean semantic token system:
- Surface hierarchy: `surface-primary` → `surface-secondary` → `surface-tertiary`
- Text hierarchy: `text-primary` → `text-secondary` → `text-tertiary`
- Semantic states: `positive`, `warning`, `negative`, `info`
- Brand scale: `brand-50` through `brand-900`
- Product-specific: `tepix-purple`

Components that respect this system look and feel unified. The sidebar, control bar, KPI cards, table headers, and separator rows all use semantic tokens correctly.

### What breaks it
1. **Recharts hex strings** — all chart colors are hardcoded. The hex values happen to match the token values but are not references to the tokens. (UI-01)
2. **`amber-*` parallel palette** — 11 sites in `bank/page.tsx` use Tailwind's built-in amber scale for the WC facility card and CTA buttons. The amber hue is semantically overlapping with `warning` but is a separate palette. (UI-07)
3. **`slate-*` in tab strip** — 3 sites. (UI-03)
4. **`bg-[#EEF7EB] border-[#BCD9B6]`** in `KPICard` — 2 hardcoded hex washes. (UI-04)
5. **`bg-green-500`** for presence dot — 1 site. (UI-08)
6. **`#6B7280`** in custom tooltip style — not in the token system at all. (UI-05)

## 4. Typography Assessment

### Strengths
- Three-font system (DM Serif Display / Inter / JetBrains Mono) is clearly defined and enforced via CSS classes.
- `font-mono` + `font-variant-numeric: tabular-nums` in `.data-cell` is correct for financial tables.
- KPI value scale (`.kpi-hero` / `.kpi-value` / `.kpi-value-compact`) is defined in CSS, not scattered across Tailwind.

### Issues
- **Six-level sub-12px scale** (`9px`, `10px`, `11px`, `12px`, `13px`, `14px`) is bespoke and untokenised. At print resolution (A4 PDF) anything below 10px is illegible. (UI-12)
- **`data-cell` vs `font-mono` inconsistency** in financial tables. (UI-16)
- **`font-display text-xl`** used directly in DSRA KPI strip instead of `.kpi-value-compact`. (UI-26)

## 5. Bank-Facing Polish Assessment

### Credible
- The term-sheet 8-cell grid is structured and data-dense appropriately.
- The "Credit Facilities Requested" sub-section with two facility cards is the clearest ask summary on the page.
- The DSCR covenant badge (PASS/FAIL) with a coloured chip is exactly what a credit analyst would look for first.
- The collateral stress table (3 tiers) is well-formatted.
- The P&L table with collapsible sections and sticky header is feature-rich.
- The ramp-year haircut callout box is a thoughtful proactive disclosure.
- `DistributionCovenantBadge` is a professional touch.

### Needs improvement
- **DSCR chart legend semantics** (UI-02 / UI-13): "Conservative" mislabelled as the realistic line is a credibility risk. A credit analyst will check the numbers and notice the mismatch.
- **3D flip card** (UI-19): The "Coming Soon" flip card with a hover perspective transform is the single most consumer/startup gesture on an otherwise institutional page. It reads as unfinished rather than as a polished teaser. A simple static "Additional materials available on request" chip would read more professionally.
- **`<h1>` is the property name at display scale** (4xl–5xl): Appropriate for a pitch deck, less appropriate for a banker credit memo. The bank view hero section (`text-4xl md:text-5xl`) is more presentation than briefing note. For the bank audience, consider reducing to a strong `text-2xl` header with a subtitle, reserving display-scale typography for the pitch deck.
- **Admin debug strip visible in bank view** (UI-09, UI-25): The "Connected viewers" section is admin-only (`isAdmin` gate) but leaks three untranslated English strings and has no `print:hidden` protection.
