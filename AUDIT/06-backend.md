## Section Analysis

### Data Model

The Firestore schema in `scenarios` is well-designed for its original purpose but has drifted as the sharing model evolved. The key structural issues are:

**What is written per schema vs reality:**

The `SavedConfiguration` type documents optional `capTable` and `waterfall` fields, but `saveConfig` and `updateConfig` never populate them. The schema advertises data that is never written, creating a false expectation on the read path. `loadConfig` does populate `capTable`/`waterfall` if they happen to be present in the loaded doc, but since they are never written, this branch is dead code for all newly saved scenarios.

**Deprecated fields actively written:**

Six `@deprecated` fields appear in scenario payloads because they live inside the `assumptions` object which is deep-cloned and written in full. `managementFee: 0` is explicitly written in the migration path. These fields are schema noise but are harmless — the engine ignores them.

**`gracePeriodYears` is written to Firestore but has no effect:**

Every scenario save includes `gracePeriodYears` for all five financing paths. The engine never reads these values. This is the ADR-0009 open debt. The UI editor allows changing these values and the new value is persisted to Firestore, but the model output does not change. This is a silent misleading UI element.

### Security & Access Control

**Firestore rules file location:** The finance platform (`villa-lev-platform`) does NOT ship its own `firestore.rules` file. The deployed rules live in the sibling `villa-lev-admin` repo at `/Users/esmacbookprom2/Desktop/Villa Lev Claude/villa-lev-admin/firestore.rules`. The finance platform's `firebase.json` has no `firestore` key — it is hosting-only. This is intentional (per comment in the rules file: "admin repo is the single source of truth") but creates an operational risk: changes to the finance app's data model require coordinating a rules deploy from a separate repository.

**Summary of bank-user Firestore access:**

| Collection | Bank user can READ? | Bank user can WRITE? |
|------------|--------------------|--------------------|
| `scenarios` (published:true) | Yes, full doc including all admin assumptions | Yes — can update or delete any scenario (BE-03, BE-19) |
| `scenarios` (published:false) | No | Only own-uid docs (no relevance since bank = anon UID) |
| `presence` | No (admin-only read) | Yes — own presence doc (but schema size mismatch breaks creates, BE-05) |
| `connectionHistory` | No (admin-only read) | Yes — own history doc |
| `seasonSnapshots` | Yes (public read) | No |
| `appConfig` | Yes (public read) | Yes — can overwrite the reference scenario pointer (request.auth != null is the only guard) |
| `users` | No | No (anonymous token has no email, invite claim path requires email) |
| `invites` | No | No |

**The admin/bank boundary is UI-only, not Firestore-enforced.** Any bank visitor who passes BankGate can read all published scenario assumptions, overwrite any scenario, and change the global reference scenario. The only Firestore-level controls are: (a) they cannot read presence/connectionHistory docs (admin-only read), and (b) they cannot write to users/invites collections. Everything else that matters for data integrity (scenarios, appConfig) is writable by any anonymous authenticated user.

### The Calculation Engine

The engine is architecturally sound for a single-operator deployment. The `viewMode: 'bank'` enforcement happens in the store (`setViewModeOverride('bank')` in bank layout's `useEffect`), which runs before any component reads model output. The Zustand store enforces this consistently — the bank layout hardwires it and cleans it up on unmount.

The bank viewMode divergence (OpCo subordinated in bank view vs senior in internal view) is correctly implemented at the engine level and tested in `src/lib/engine/__tests__/model.viewMode.test.ts`.

**Client-side tamper risk (BE-07):** Because this is a fully static export with no server-side computation, a technically sophisticated external user with browser DevTools could mutate the Zustand store and produce different numbers. This is not a Firestore or rules problem — it is an inherent property of a client-side financial model. No amount of Firestore hardening mitigates this. The only mitigations are output-level (watermarks, hashes, "generated by" footers).

### Integrations & APIs

**`useEuribor` (ECB API):**
- Source: `https://data-api.ecb.europa.eu/service/data/FM/B.U2.EUR.RT0.BB.B.A.A.TA.N?lastNObservations=1&format=jsondata`
- Caching: 6-hour TTL in sessionStorage — per-tab, not cross-tab
- Failure mode: `status: 'error'`, `rate: null`. The page shows a banner and falls through to the model's stored `euriborRate`. Graceful degradation is implemented.
- Risk: the stored `euriborRate` in assumptions reflects the last time an admin set it — could be weeks stale. The bank page shows it as the "effective rate" with no indication of when it was last updated.
- Cross-origin: the ECB API does not require credentials, but CORS must allow `villa-lev-finance.web.app`. This is a runtime dependency; if ECB changes CORS policy, the hook silently degrades to the stored rate.

**`useSeasonSnapshot` (Firestore cross-project read):**
- This is NOT actually cross-project. Both apps share Firebase project `villa-lev-admin`. The read is a standard Firestore `onSnapshot` on `seasonSnapshots/latest` within the same project.
- The `seasonSnapshots` rules explicitly allow public read (`allow read: if true`) to support unauthenticated bank/investor visits.
- Failure mode: document missing → static fallback from `currentVillaActuals.ts`. The admin layout shows a "Stale snapshot" banner. Bank/pitch views also use `useSeasonSnapshot` but may not show the same staleness indicator.
- Availability: if the sibling ops app has never published a snapshot (or the doc is deleted), every view falls back to the static actuals silently. No alerting outside the admin banner.

### What Would Most Improve Trust and Security

In priority order:

1. **Stop emitting `NEXT_PUBLIC_ADMIN_PASS` in the bundle (BE-01).** Move the password check server-side via a Firebase Custom Token or Firebase App Check. This is the single highest-impact change. The current architecture makes the admin password trivially readable by any visitor who opens DevTools.

2. **Restrict scenario delete to owner (BE-19).** One-line rule change. Closes the "any bank visitor can delete all scenarios" hole with zero functional impact on legitimate use.

3. **Fix the presence schema size mismatch (BE-05).** Change `size() == 8` to `size() == 9` in the rules to include the `actions: []` field, or drop `actions` from the create payload. Without this fix, the `/admin/connections` board has no presence data for any bank visitor, and all presence writes silently fail.

4. **Audit published scenarios for internal-only data (BE-02).** Before the next bank meeting, verify which scenarios are `published: true` and whether their `assumptions` blobs contain fields that should not be visible to a bank user (e.g. internal cap table targets, unreported debt terms, stress test overrides).

5. **Add output watermarks and model hash to bank exports (BE-07).** No code architecture change required — add a footer to `exportBankPresentation.ts`. Cheap, trust-building, and appropriate for TEPIX III submission materials.
